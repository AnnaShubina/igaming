Case 1

Original crash game Apollo

We developed a unique white-label game, Apollo, from scratch, and we have successfully integrated this into the gaming sites of betting partners. The game’s development process was complex, and it included several blocks of work, including mathematics, animation + interface, operator API, back office (reporting on rates, settings, and bonuses). The main advantage of the game is the possibility of full customization. For iGaming operators, the game is completely flexible.
Technologies
Java (Play), PostgreSQL
Term
4 months
Outcome
The brand-new game will allow our partners to expand their portfolios of gaming products, enhancing the gaming experience and increasing their GGR.
Case 2

Our very own casino bonus program

We needed to develop our own casino bonus program from scratch, as a separate microservice based on the Spring framework. This feature is integrated into the overall system and supports interaction with other components through RabbitMQ message queues, which are an external event bus.
Technologies
Java (Play), PostgreSQL
Term
3 months
Outcome
We developed a new feature that allows us to credit bonuses to players based on various user actions that are completed, for example, the first deposit bonus, specified deposit amount bonuses, a Visa/MasterCard deposit bonuses, split bonuses, multibet bonuses, 10% cash back bonuses, a bonus for regular active players on their birthday etc.
Case 3

Automated systems for analysing incidents of interrupted bets

Customers would often come across obstacles and friction points when making money transfers between their wallet and their gaming account. Our task was to develop and integrate a module for identifying and alleviating interruption rates for the casino’s integration and their wallet service because manual analysis of emerging errors tended to be very time-consuming and resource- consuming.
Technologies
Java (Play), PostgreSQL
Term
2 months
Outcome
The new system will alleviate these issues by ensuring the uninterrupted flow of the money transfer system, even with an ever-growing number of users.
Case 4

Accounting for rates and payments - protection against double charges

One of the common bugs in highly loaded systems is double money charges/debits from users. This occurs thanks to the parallel processing of the same bet request. We have developed a system for checking limits when processing a transaction via cutting-edge optimistic locking.
Technologies
Java (Play), PostgreSQL
Term
1 month
Outcome
We managed to operate the new system uninterrupted, providing a clear record of rates and payments and excluding any duplication of charges.
